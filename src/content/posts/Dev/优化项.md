# 优先级总结

| 优先级 | 问题 | 预期收益 |
|--------|------|----------|
| 高 | 提取并缓存 getStorageConfig | 减少 DB 查询，每次上传少一次全表扫描 |
| 高 | extractDominantColors 与 storage.upload 并行 | 减少单张上传延迟 |
| 高 | 相册关联改 Promise.all | 多相册时减少等待时间 |
| 中 | 预览图提升到 state 层 | 避免拖拽重排时重复生成 |
| 中 | GPS strip 改并行 | 多文件时减少预处理时间 |
| 低 | 队列调度优化 | 减少 50ms 固定延迟 |

- - 🔴 高优先级
    1. useEditorState 选择器过于宽泛，频繁触发重渲染
    - 位置: NarrativeTipTapEditor.tsx L586-655
    - 问题: selector 返回包含 20+ 属性的完整对象，任何编辑器操作（打字、移动光标）都会导致整个 toolbar 重渲染。
    - 建议: 拆分为多个细粒度 useEditorState 调用，每个只订阅需要的属性；或使用 useMemo + JSON.stringify 做浅比较。
    2. onUpdate 每次击键都调用 onChange，无防抖
    - 位置: NarrativeTipTapEditor.tsx L543-547
    - 问题: 每次内容变化都同步触发 onChange(html)，父组件 StoryEditorView 中 setCurrentStory 也跟着每次击键执行。对于长文档，getHTML() 序列化成本很高。
    - 建议: 在编辑器内部加 200-300ms 防抖，或者使用 onTransaction + 节流策略。
    3. processedContent 的 Markdown 检测在每次 value 变化时全量扫描
    - 位置: NarrativeTipTapEditor.tsx L224-272, L311-327
    - 问题: convertMarkdownToHtml 用多个正则逐行扫描，isMarkdownContent 对 11 种模式做全文件匹配。大文档时性能差。
    - 建议: 只在首次加载时检测；后续 setValue 时由调用方标记内容类型。
    4. NarrativeTipTapEditor 单文件过大 (1734 行)
    - 问题: 一个文件包含编辑器初始化、toolbar 按钮、颜色选择器、AI 集成、Markdown 转换、imperative handle 等所有逻辑。
    - 建议: 拆分为：
      - useNarrativeEditor.ts — 编辑器实例 + 扩展配置
      - EditorToolbar.tsx — 工具栏 UI
      - ColorPickerMenu.tsx — 文字/背景色选择器
      - markdown-converter.ts — Markdown ↔ HTML 转换
      - useEditorImperativeHandle.ts — imperative API
    5. scaleLastImage 遍历整个文档查找最后一个图片
    - 位置: NarrativeTipTapEditor.tsx L853-892
    - 问题: state.doc.descendants() 遍历所有节点，对长文档有性能问题。
    - 建议: 缓存最后插入的图片位置，或使用 ProseMirror 的索引结构优化查找。
    ---
    🟡 中优先级
    6. 颜色选择器渲染大量 DOM 节点
    - 位置: L1513-1533, L1651-1671
    - 问题: basic + more 两个 tab 共 64 个颜色按钮，每次打开都完整渲染。
    - 建议: 使用 React.memo 包裹单个颜色按钮；或虚拟列表。
    7. useImperativeHandle 依赖数组过大，频繁重建
    - 位置: L783-894
    - 问题: imperativeHandle 依赖 [editor, focusEditor, insertInlineImage, onChange]，其中 onChange 如果父组件未 memo 化会导致每次渲染都重建 handle。
    - 建议: 确认父组件传入的 onChange 已 useCallback 化；或改用 ref 存储方法。
    8. AI 助手 selectionUpdate 事件同步上下文
    - 位置: L720-728
    - 问题: 每次光标移动都触发 syncAiSelectionState，内部调用 doc.textBetween() 三次（选中文本、上下文前、上下文后），对大文档有开销。
    - 建议: 加节流（100-200ms），AI 面板打开时才同步。
    9. PastedStyleMark 和 PastedBlockStyle 的 sanitizeStyleValue 逻辑重复
    - 位置: 两个文件中完全相同的函数
    - 建议: 提取到共享 sanitize-style.ts 模块。
    10. ImageGroup 的 handleDrop 逻辑复杂且边界情况多
    - 位置: ImageGroup.ts L48-165
    - 问题: 嵌套多层检查（imageGroup → standalone image → parent level），代码可读性差，moved 参数检查存在逻辑漏洞（L55 if (moved) return false 与后续 L84 if (moved) 矛盾）。
    - 建议: 重构为状态机模式，明确处理每种 drop 场景。
    11. 缺少键盘快捷键
    - 问题: 没有 Ctrl/Cmd+S 保存、Ctrl/Cmd+K 插入链接等常见快捷键。
    - 建议: 通过 addKeyboardShortcuts 扩展或 editorProps.handleKeyDown 添加。
    ---
    🟢 低优先级 / 体验优化
    12. shouldRerenderOnTransaction: true
    - 位置: L542
    - 问题: 每次 transaction 都触发 ProseMirror 重渲染，包括不改变视图的操作（如 selection 变化）。
    - 建议: 设为 false，除非有特定需求。
    13. 图片拖拽调整大小时缺少视觉反馈
    - 位置: ResizableImage.ts
    - 问题: 拖拽过程中没有尺寸提示（如显示当前宽高）。
    - 建议: 添加浮动尺寸标签。
    14. convertMarkdownToHtml 不支持代码块、水平线等
    - 位置: L224-272
    - 问题: 正则实现的 Markdown 转换器不支持 fenced code blocks、horizontal rules、task lists 等。
    - 建议: 使用 turndown 或 markdown-it 替代手写正则。
    15. ensureFirstParagraphHasDropCap 在每次 processedContent 变化时执行
    - 位置: L581-584
    - 问题: 即使 dropCap 已设置，每次内容更新都会重新检查。
    - 建议: 添加守卫条件，只在文档首个段落变化时执行。
    16. 缺少内容自动保存的视觉指示器
    - 问题: 管理后台 StoryEditorView 有 draft 状态，但 QuickStoryEditor 使用 textarea 而非 TipTap 编辑器，两者体验不一致。
    - 建议: 统一使用 TipTap 编辑器，并添加保存状态指示。
    ---
    三、风险项
  
    风险
    shouldRerenderOnTransaction: true 改为 false 可能导致某些扩展不更新
    onUpdate 加防抖可能丢失最后一次编辑
  
    拆分大文件可能引入 import 循环
    ---
    四、建议实施顺序
    1. 先修性能: useEditorState 拆分 → onUpdate 防抖 → shouldRerenderOnTransaction: false
    2. 再修质量: 拆分大文件 → 提取重复代码 → 修复 ImageGroup drop 逻辑
    3. 最后加功能: 键盘快捷键 → Markdown 转换器升级 → 拖拽尺寸反馈
